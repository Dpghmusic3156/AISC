# Plan 4.2: App Wiring & Loading State

## Objective
Wire the new `ResultWindow` into the main application capture flow so that users see a Loading window during the API wait, followed by the actual markdown result window, eliminating the ugly synchronous `MessageBox` calls.

## Context
- src/AIScreenCapture.UI/App.xaml.cs

## Tasks

<task type="auto">
  <name>Create Loading UI State</name>
  <files>
    src/AIScreenCapture.UI/Views/LoadingWindow.xaml
    src/AIScreenCapture.UI/Views/LoadingWindow.xaml.cs
  </files>
  <action>
    Create a small, borderless `LoadingWindow` with a modern MaterialDesign circular progress spinner and text like "Analyzing image...".
    - `WindowStyle="None"`, `Topmost="True"`, `AllowsTransparency="True"`, `WindowStartupLocation="CenterScreen"`
  </action>
  <verify>dotnet build src/AIScreenCapture.UI</verify>
  <done>LoadingWindow exists and builds successfully.</done>
</task>

<task type="auto">
  <name>Update Capture Flow</name>
  <files>src/AIScreenCapture.UI/App.xaml.cs</files>
  <action>
    Modify the `OnRegionSelected` method to use the new UI components instead of `MessageBox`:
    - Before calling `client.SendImageAsync(...)`, instantiate and `Show()` the `LoadingWindow`.
    - Also update the logic so API clients are invoked gracefully in a `try/catch`. 
    - Once the `await` finishes, `Close()` the loading window.
    - Instantiate the `ResultWindow`, call `SetResult(response, activePreset.ModelName)`, and `Show()` it.
    - If an error occurs (timeout, invalid key), open the `ResultWindow` (or a `MessageBox` fallback) with the error text.
  </action>
  <verify>dotnet build src/AIScreenCapture.UI</verify>
  <done>App.xaml.cs references LoadingWindow and ResultWindow without any MessageBox.Show() calls for the successful flow.</done>
</task>

## Success Criteria
- [ ] Capturing a region instantly shows a central loading spinner.
- [ ] Once the API responds, the loading window disappears.
- [ ] The `ResultWindow` appears with the formatted AI output.
